# Архитектурное решение по логированию

## 1. Анализ системы и необходимые логи

На основе C4-диаграммы и бизнес-процессов компании «Александрит» определены следующие источники логов:

- **Shop API** (Spring Boot): создание и изменение заказов, загрузка 3D-файлов, запросы к БД, вызовы RabbitMQ.
- **CRM API** (Spring Boot): работа с клиентами, подтверждение заказов, интеграция с MES через очередь.
- **MES API** (C#): расчёт стоимости, назначение на операторов, изменение статусов производства.
- **Messages Queue** (RabbitMQ): события публикации/потребления, ошибки маршрутизации, dead-letter.
- **Shop DB / MES DB** (PostgreSQL): долгие запросы, ошибки подключений (логируются на уровне приложений).
- **3D files storage** (S3): операции загрузки/чтения, ошибки доступа.

**Логи уровня INFO, которые необходимо собирать:**

1. **Жизненный цикл заказа**:
   - Создание заказа: `order_id`, `customer_id`, `timestamp`, `source` (B2C/B2B), `status=INITIATED`
   - Изменение статуса: `order_id`, `old_status`, `new_status`, `timestamp`, `initiator` (user/operator/system)
   - Расчёт стоимости: начало/окончание, `order_id`, `duration_ms`, `polygons_count`
   - Отправка/получение сообщения в RabbitMQ: `order_id`, `routing_key`, `message_id`, `timestamp`
   - Запросы к внешним API (S3, другие сервисы): `url`, `method`, `status_code`, `duration_ms`

2. **Технические события**:
   - Запуск/остановка сервиса: `service_name`, `version`, `timestamp`
   - Медленные запросы к БД: `query`, `duration_ms`, `tables`
   - Ошибки валидации входных данных: `field`, `value`, `error`

3. **События безопасности**:
   - Неуспешная аутентификация/авторизация: `user`, `ip`, `endpoint`
   - Изменение критических настроек: `parameter`, `old_value`, `new_value`

**Использование других уровней**:
- **DEBUG/TRACE** – в dev-окружении для детальной отладки (запросы, ответы, внутренние состояния).
- **WARN** – повторные попытки операций (retries), превышение мягких лимитов (число подключений).
- **ERROR** – исключения, падения, ошибки интеграции, потерянные сообщения.
- **FATAL** – критические сбои, приводящие к остановке сервиса.

## 2. Мотивация

Внедрение централизованного логирования критически важно для «Александрит» по следующим причинам:

- **Сокращение времени расследования инцидентов (MTTR)** – вместо ручного сбора логов с разных инстансов команда сможет мгновенно найти все события по конкретному заказу.
- **Проактивное обнаружение проблем** – настройка алертов на аномалии (всплески ошибок, отсутствие ожидаемых событий) позволит реагировать до жалоб клиентов.
- **Снижение нагрузки на поддержку** – специалисты смогут самостоятельно исследовать историю заказа, не отвлекая разработчиков.
- **Прозрачность бизнес-процессов** – дашборды с динамикой статусов заказов, времени расчёта и количества ошибок дадут бизнесу объективную картину.

**Ожидаемые метрики улучшения**:
1. Среднее время восстановления (MTTR) снизится с часов до минут.
2. Доля инцидентов, обнаруженных по логам до жалоб клиентов, вырастет с 0% до 80%.
3. Количество обращений в поддержку по проблемам с заказами сократится на 50%.
4. Время поиска причины сбоя уменьшится на 70%.
5. Процент заказов, завершившихся с ошибками, станет измеримым и контролируемым.

**Приоритеты внедрения**:
- **Первая очередь**: Shop API, MES API – ключевые сервисы обработки заказов и расчёта стоимости. Без логов этих систем невозможно расследовать большинство инцидентов.
- **Вторая очередь**: CRM API, RabbitMQ – работа с клиентами и интеграционный слой.
- **Третья очередь**: фронтенды (браузерные логи через Sentry/ELK), базы данных (расширенное логирование).

Трейсинг уже покрывает эти сервисы, но логирование необходимо для детального понимания бизнес-событий и ошибок, которые не всегда видны в трейсах.

## 3. Предлагаемое решение

### 3.1. Технологический стек

- **Сбор логов**: Filebeat (легковесный агент) на каждом EC2‑инстансе приложений, отправляющий логи в централизованный коллектор.
- **Агрегация**: Logstash для парсинга, маскирования чувствительных данных, обогащения метаданными (имя сервиса, окружение).
- **Хранилище**: OpenSearch в виде managed‑сервиса Yandex Cloud Managed Service for OpenSearch. Обеспечивает совместимость с Elasticsearch API, привычный Kibana‑интерфейс, автомасштабирование и резервирование.
- **Визуализация и анализ**: Kibana (идет в комплекте с OpenSearch) для поиска, дашбордов и алертинга.
- **Алертинг**: ElastAlert или встроенные механизмы OpenSearch Alerting.

### 3.2. Интеграция в приложения

- **Java (Shop API, CRM API)**: Logback + JSON encoder + Filebeat sidecar.
- **C# (MES API)**: Serilog с провайдером для Elasticsearch/OpenSearch.
- **Все сервисы** обогащают логи стандартными полями: `service`, `environment`, `trace_id`, `order_id` (из контекста трейсинга).
- **Структура лога**: JSON с фиксированным набором полей для удобного парсинга.

### 3.3. Топология развёртывания (новые компоненты)

На C4-диаграмме контейнеров были добавлены:

- **Filebeat** (sidecar‑контейнер на каждом узле приложений) – сбор логов stdout/файлов и отправка в Logstash.
- **Logstash** – отдельный сервис для обработки и маршрутизации логов.
- **OpenSearch Cluster** – managed‑сервис в облаке.
- **Kibana** – дашборд для поиска и визуализации.

Все новые компоненты выделены **красным цветом** на прилагаемой в данном каталоге диаграмме jewerly_c4_model.drawio.

### 3.4. Политика безопасности

- **Маскирование чувствительных данных**: на уровне Logstash применяются фильтры, заменяющие или удаляющие поля, содержащие ФИО, email, номера телефонов, пароли. Все `customer_id` хешируются, если не требуются для бизнес-анализа.
- **Контроль доступа**: Kibana интегрирована с корпоративной системой аутентификации (Keycloak/Yandex ID). Ролевая модель:
  - `viewer` – доступ только к дашбордам без возможности поиска по сырым данным (поддержка, продакт-менеджер).
  - `developer` – поиск по логам dev/staging, ограниченный доступ к prod (только ERROR/WARN без персональных данных).
  - `admin` – полный доступ к настройкам и всем данным.
- **Шифрование**: трафик между компонентами по TLS, хранилище OpenSearch с шифрованием at rest.

### 3.5. Политика хранения

- **Отдельные индексы** для каждого приложения и окружения (например, `shop-api-prod-06.07.2026`).
- **Ротация**: ежедневно, закрытие индекса через 1 день, удаление через 30 дней для hot‑узлов.
- **Архивация**: cold‑тир на S3 с хранением до 90 дней для аудита и бизнес-аналитики.
- **Лимиты размера**: максимальный размер индекса 50 ГБ, количество шардов подбирается исходя из суточного объёма (планируется не более 20 ГБ/сутки на старте).

### 3.6. Превращение системы сбора логов в систему анализа

- **Алертинг**:
  - Количество ERROR‑логов за 5 минут > 10 → уведомление в корпоративный мессенджер, автоматическое создание тикета в Jira.
  - Отсутствие логов о создании заказов за последние 15 минут (для рабочего времени) → критический алерт дежурному.
  - Рост количества WARN‑ов по повторным попыткам → предупреждение о возможной деградации.

- **Поиск аномалий**:
  - Использование встроенного OpenSearch Anomaly Detection на основе машинного обучения для автоматического выявления отклонений в количестве заказов, ошибок, времени ответа.
  - Пример: внезапный рост числа запросов к API в нерабочее время может указывать на DDoS-атаку; автоматически отправляется алерт и запускается инцидент безопасности.

- **Визуализация**:
  - Дашборд «Здоровье заказов» – распределение статусов, время нахождения в каждом статусе, процент успешно завершённых.
  - Дашборд «Производительность» – время ответа API, медленные запросы, ошибки.
  - Дашборд «Очереди» – глубина очередей, сообщения в DLQ.

## 4. Дополнительное задание: критерии выбора технологии

| Критерий | ELK (Elasticsearch) | OpenSearch | Grafana Loki | Splunk |
|----------|---------------------|------------|--------------|--------|
| **Лицензия** | Elastic License (несвободная, ограничения) | Apache 2.0 (полностью открытая) | AGPLv3 (открытая, но требует раскрытия кода при модификации) | Проприетарная (дорогая) |
| **Совместимость с облаком** | Есть в Yandex Cloud Managed Service, но с ограничениями из-за лицензии | Yandex Cloud Managed Service for OpenSearch (полнофункциональный) | Нет managed-сервиса, требует самостоятельной установки | Splunk Cloud доступен, но высокая цена |
| **Возможности поиска** | Отличный полнотекстовый поиск, агрегации | Полная совместимость с Elasticsearch 7.x API | Ограниченный поиск по меткам, нет полнотекстового анализа сообщений | Мощный SPL, но специфичный |
| **Экосистема** | Kibana, Beats, Logstash – зрелая экосистема | Kibana (форк OpenSearch Dashboards), совместимость с Beats | Grafana, Promtail, Loki – легковесная, хорошо интегрируется с остальной наблюдаемостью | Собственная экосистема приложений |
| **Стоимость владения** | Бесплатно для базовых функций, платные фичи (Alerting, ML) | Полностью бесплатно, включая Alerting и Anomaly Detection | Бесплатно, но требует больше ручной настройки | Очень высокая стоимость лицензий и инфраструктуры |
| **Простота эксплуатации** | Managed-сервис в облаке, но сложное обновление из-за лицензии | Managed-сервис полностью управляемый, проще в миграции | Требует развёртывания и поддержки кластера | Managed-сервис, но требует сертифицированных специалистов |
| **Поддержка формата логов** | Широкая, любые структурированные логи | Аналогично Elasticsearch | Оптимизирован для логов в формате с лейблами (JSON, key-value) | Широкая, но часто требует настройки |

**Выбор: OpenSearch** – оптимальный компромисс для «Александрит»:
- Открытая лицензия исключает юридические риски и дополнительные расходы.
- Managed‑сервис в Yandex Cloud снижает нагрузку на DevOps.
- Полная совместимость с привычным инструментарием (Kibana, Beats), что важно для текущей экспертизы команды.
- Встроенные средства алертинга и обнаружения аномалий позволяют сразу строить систему анализа, не докупая платные модули.
- Стоимость прогнозируема и масштабируется вместе с ростом объёмов логов.

**Альтернативы**:
- **Loki** – хорош для логов микросервисов, если основная цель – быстрый поиск по меткам, но отсутствие полнотекстового поиска по содержимому сообщений может усложнить отладку бизнес‑логики.
- **Splunk** – слишком дорог для компании масштаба «Александрит» и требует обучения персонала.
- **ELK** – нежелателен из-за лицензионных изменений, которые могут привести к невозможности использования облачных сервисов без коммерческой лицензии.
