# Выбор и настройка мониторинга в системе

## 1. Мотивация

В текущем состоянии мониторинг «Александрит» ограничен инструментами веб-аналитики (Яндекс.Метрика), которые покрывают только поведение пользователей на фронтенде интернет-магазина. После открытия API для B2B-партнёров и интеграции внутренних систем через RabbitMQ возникли принципиально новые потоки взаимодействия, невидимые для существующих средств наблюдения.

Проблемы, с которыми столкнулся бизнес (просроченные заказы, пропажа сообщений, медленная работа интерфейса MES, потеря B2B-контрактов), проявляются постфактум — через жалобы клиентов, операторов и партнёров. Отсутствие системного мониторинга не позволяет:

- оперативно обнаруживать аномалии до того, как они повлияют на пользователей;
- локализовать узкие места (база данных, очередь, API, файловое хранилище);
- оценивать влияние изменений и роста нагрузки;
- выполнять данные обещания по SLA (расчёт стоимости за 2–3 минуты, своевременная обработка заказов).

Внедрение технического мониторинга даст:

1. **Проактивное выявление инцидентов.** Алерты о росте очереди dead-letters, тайм-аутах или увеличении задержек позволят реагировать до потери заказов.
2. **Прозрачность для принятия решений.** Графики загрузки CPU, памяти и соединений БД покажут, когда именно необходимо масштабирование, а не «когда всё упало».
3. **Снижение времени расследования (MTTR).** Корреляция событий в RabbitMQ с ошибками HTTP 500 и задержками конкретного API даст возможность быстро найти причину.
4. **Бизнес-уверенность.** Руководство сможет видеть реальную производительность системы в разрезе B2B/B2C и убедиться, что инвестиции в архитектуру (например, асинхронный расчёт стоимости) приносят эффект.

Таким образом, мониторинг — это не техническая прихоть, а инструмент защиты выручки и репутации компании, особенно критичный в условиях кратного роста заказов.

## 2. Выбор подхода к мониторингу

Для разных компонентов системы целесообразно применять комбинацию методологий:

- **API‑интерфейсы (Shop API, CRM API, MES API):** **RED** (Rate, Errors, Duration) — фокус на запросах, которые инициируют пользователи и внешние системы. Этот подход напрямую отражает пользовательский опыт.
- **Серверные ресурсы:** **USE** (Utilization, Saturation, Errors) — для раннего обнаружения нехватки CPU, памяти, сетевой пропускной способности.
- **Базы данных и очереди сообщений:** **Четыре золотых сигнала** (Latency, Traffic, Errors, Saturation) — наиболее показательная методология для отслеживания корректности потока данных.
- **Бизнес‑события:** отдельная группа метрик, основанная на событиях жизненного цикла заказа (например, время перехода между статусами, количество заказов в статусе `PRICE_CALCULATED` дольше X минут). Это не классический RED/USE, но критично для предметного мониторинга.

Такой гибридный подход гарантирует, что мы контролируем и техническое здоровье, и бизнес‑потоки.

## 3. Метрики и их обоснование

Из предложенного списка и с учётом специфики системы выбраны следующие метрики. Для каждой указаны цель и необходимые ярлыки (labels) для группировки.

### 3.1. Очереди сообщений RabbitMQ

| Метрика | Зачем нужна | Ярлыки |
| :--- | :--- | :--- |
| **Number of dead-letter-exchange letters in RabbitMQ** | Прямой индикатор «потерянных» заказов: сообщения, не обработанные ни одним потребителем из‑за ошибок формата, исключений или длительного цикла обработки. Без этой метрики компания узнаёт о пропавшем заказе только из звонка партнёра. | `queue_name`, `error_type` (например, `deserialization_error`, `business_logic_error`), `source_system` (MES, CRM, Shop) |
| **Number of messages in flight in RabbitMQ** | Показывает текущую глубину очереди и загруженность обработчиков. Рост очереди без увеличения потребителей сигнализирует о деградации или блокировке сервиса. | `queue_name`, `consumer_group` (если применимо) |

Также предлагается добавить ещё одну критичную метрику RabbitMQ:

- **Message processing latency** (время от публикации до подтверждения обработки). Измеряется в разрезе `queue_name`, `message_type` (например, `OrderPriceCalculationRequested`). Позволяет зафиксировать фактические нарушения SLA по расчёту стоимости.

### 3.2. API‑интерфейсы (RED + сигналы насыщенности)

Для каждого из трёх API (Shop, CRM, MES) отслеживаем:

| Метрика | Зачем нужна | Ярлыки |
| :--- | :--- | :--- |
| **Number of requests (RPS)** | Показывает нагрузку в разрезе интерфейсов. Без разделения невозможно понять, растёт ли трафик за счёт B2B или B2C. | `api_name` (shop/crm/mes), `endpoint` (например, `/api/orders`, `/api/price`), `http_method`, `user_type` (b2c, b2b, internal) |
| **Response time (latency)** (гистограмма/процентили) | Отражает пользовательский опыт. Важно видеть медиану (p50) и «длинный хвост» (p95, p99), потому что расчёт стоимости может выбиваться из общего потока. | `api_name`, `endpoint`, `http_method`, `status_code_class` (2xx, 4xx, 5xx) |
| **Number of HTTP 500 (errors rate)** | Индикатор отказов сервера, прямо влияющих на успешность оформления заказов. | `api_name`, `endpoint`, `exception_type` (если доступен) |
| **Number of HTTP 200** | Необходим для вычисления доли ошибок (error budget) и подтверждения, что трафик обрабатывается успешно. | `api_name`, `endpoint` |
| **CPU % / Memory Utilisation** | Показывают насыщенность ресурсов сервера. Рост загрузки CPU при неизменном RPS может указывать на неэффективный код или атаку. | `instance_id`, `api_name` |
| **Kb transferred (received/sent)** | Мониторинг сетевого трафика помогает выявить аномально большие запросы (например, загрузка 3D‑моделей) и потенциальные узкие места в канале. | `api_name`, `direction` (in/out) |

**Дополнительные метрики API:**

- **Number of concurrent requests** (или `Number of simultaneous sessions`). Аналогична из списка, но более детальная: количество активных соединений/потоков в конкретный момент. Ярлыки: `api_name`, `endpoint`. Помогает оценить, насколько сервер близок к исчерпанию пула потоков, что критично для синхронных операций.

### 3.3. Базы данных (Shop DB, MES DB)

| Метрика | Зачем нужна | Ярлыки |
| :--- | :--- | :--- |
| **Number of connections** | Показывает, сколько клиентских соединений активно. Выход на плато максимальных коннектов приводит к отказам в подключении. Особенно важно для `MES db` при медленных запросах к дашбордам. | `db_name`, `api_name` (источник), `read/write` |
| **Memory Utilisation for db instance** | Индикатор того, насколько эффективно используется буферный кэш. Недостаток памяти заставляет БД чаще обращаться к диску, что резко увеличивает время запросов, вызывая задержки. | `db_name`, `instance_id` |
| **Size of db instance** | Контроль за темпами роста данных. Неожиданное увеличение может быть признаком утечки или неочищаемых таблиц. | `db_name` |

**Дополнительно** метрики на уровне СУБД:

- **Replication lag (секунды)** — для будущей read‑реплики. Нарушение лага приводит к рассинхронизации данных на дашбордах (списки заказов не актуальны).
- **Deadlocks / Lock waits per second** — прямо указывает на блокировки, которые могут замедлять дашборд операторов.

### 3.4. Инфраструктура и хранилище

| Метрика | Зачем нужна | Ярлыки |
| :--- | :--- | :--- |
| **Size of S3 storage** | Отслеживание роста объёма 3D‑файлов. Влияет на стоимость и политику хранения. | `bucket_name` |

**Дополнительно:**

- **S3 request latency** — прямая загрузка и чтение файлов напрямую влияют на время оформления заказа и расчёт.

### 3.5. Бизнес‑метрики (выходящие за рамки технического мониторинга, но необходимые)

Эти метрики не из предложенного списка, но критичны для предметной наблюдаемости:

- **Orders stuck in status** (`PRICE_CALCULATED`, `SUBMITTED`) > N минут. Ярлыки: `status`, `order_type` (b2c/b2b). Позволяет немедленно обнаружить, что заказы не переходят в производство.
- **Price calculation duration distribution** (гистограмма по заказам). Прямое измерение времени расчёта, разбитое по сложности модели. Покажет, соблюдается ли обещанное время 2–3 минуты и какие модели вызывают 30‑минутные расчёты.

## 4. План действий (высокоуровневый)

### 4.1. Выбор технологического стека

- **Сбор метрик приложений:** Spring Boot Actuator + Micrometer (Prometheus registry) для Java‑сервисов (Shop API, CRM API). Для C#‑приложения (MES API) — библиотека `prometheus-net` либо экспорт через OpenTelemetry .NET SDK.
- **Сбор инфраструктурных метрик:** Node Exporter на каждом EC2‑инстансе (CPU, память, диск, сеть).
- **Экспортёры для БД и очередей:** встроенный exporter для PostgreSQL (например, `postgres_exporter`), RabbitMQ Prometheus exporter.
- **Хранение временных рядов:** Prometheus.
- **Визуализация:** Grafana (развернуть рядом с Prometheus).
- **Алертинг:** Alertmanager, интегрированный с корпоративным мессенджером.

### 4.2. Этапы реализации

1. **Развернуть инфраструктуру мониторинга:** создать инстанс для Prometheus и Grafana, настроить сетевую доступность до всех EC2 и managed‑сервисов.
2. **Установить экспортёры:** настроить Node Exporter на все инстансы приложений, подключить `postgres_exporter` к Shop DB и MES DB, включить RabbitMQ Prometheus плагин.
3. **Инструментировать приложения:** добавить micrometer‑зависимости и эндпоинт `/actuator/prometheus` в Shop API и CRM API; в MES API добавить метрики для эндпоинтов и длительности операций расчёта.
4. **Сконфигурировать дэшборды Grafana:** создать отдельные дашборды для каждого API (RED), RabbitMQ, баз данных, обзорный бизнес‑дашборд.
5. **Настроить алерты:** определить правила для критических состояний и протестировать каналы уведомлений.

## 5. Показатели насыщенности и пороговые значения (Дополнительное задание)

Насыщенность (saturation) показывает, насколько близок ресурс к исчерпанию. Пороговые значения выбираются на основе нагрузочного тестирования и исторических данных, но предварительно можно использовать следующие безопасные границы.

| Ресурс | Метрика насыщенности | Порог (warning / critical) | Действия при превышении |
| :--- | :--- | :--- | :--- |
| **RabbitMQ** | `Number of messages in flight` (глубина очереди `price_calculation_requests`) | Warning: > 50 сообщений; Critical: > 200 | **Warning:** автоматический тикет на команду, проверка потребителей. **Critical:** уведомление в мессенджер техлида + дежурного, немедленное добавление экземпляров воркеров. |
| **RabbitMQ** | `Number of dead-letter-exchange letters` | Любое ненулевое значение > 0 за 5 мин | **Critical:** немедленный алерт в канал поддержки. Ручной разбор сообщений, повторная публикация. |
| **Shop API / CRM API** | `CPU %` | Warning: > 70% устойчиво 10 мин; Critical: > 90% | **Warning:** тикет на оптимизацию, проверка на утечку потоков. **Critical:** ручной вертикальный апгрейд или горизонтальное масштабирование (добавление инстанса). |
| **MES API** | `CPU %` и `Response time p95` | Latency p95 > 5 сек или CPU > 80% | **Warning:** проверка фоновых задач расчёта. Если связано с синхронными расчётами, повышение приоритета по переходу к асинхронной модели. |
| **PostgreSQL (MES db)** | `Number of connections` (% от max_connections) | Warning: > 70%; Critical: > 85% | **Warning:** анализ долгих транзакций, автоматическая рассылка разработчикам. **Critical:** принудительное отключение неактивных сессий, добавление реплики для чтения. |
| **MES DB** | `Memory utilisation` (общий объем RAM инстанса) | Warning: > 85% утилизации всего инстанса | Указывает на нехватку shared_buffers. Тикет на увеличение ресурсов или оптимизацию запросов. |
| **Shop DB** | `Replication lag` (после внедрения реплик) | Warning: > 2 сек; Critical: > 10 сек | **Warning:** тикет. **Critical:** автоматическое исключение реплики из балансировки чтения, уведомление администратору БД. |
| **3D‑файлы (S3)** | `Size of S3 storage` | Warning: > 80% выделенной квоты | Тикет на очистку или расширение хранилища. |д
| **Бизнес‑насыщенность** | `Orders stuck in PRICE_CALCULATED > 30 min` | Любое количество заказов > 0 | **Critical:** немедленное оповещение в чат производства и техподдержки, проверка очереди расчёта стоимости. |

Все алерты уровня Critical должны быть настроены с эскалацией: если дежурный не подтвердил получение в течение 5 минут, сообщение дублируется всей команде и тимлиду.

**Автоматическая реакция:** Там, где инфраструктура позволяет (например, в облаке Yandex Cloud с группами автоскейлинга), пороги по CPU и глубине очереди могут быть завязаны на автоматическое добавление инстансов приложений или воркеров. Однако для MES API, работающего с состоянием, автоматическое масштабирование должно быть согласовано с архитектурой (сессии, блокировки), поэтому первоначально — ручное расширение с последующей автоматизацией.
