# Архитектурное решение по трейсингу

## 1. Анализ системы и мест, где заказ может «сломаться» или зависнуть

На основе C4-диаграммы и описания бизнес-процессов определены следующие точки потенциальной потери или задержки заказа:

1. **Internet Shop → Shop API** — запрос на создание заказа может упасть из-за сетевого сбоя или невалидных данных. Без сквозного контекста сложно понять, был ли заказ сохранён в `Shop DB` и ушло ли событие в очередь.
2. **Shop API → Messages Queue (RabbitMQ)** — публикация события `OrderSubmitted` может не состояться. Если контекст не проброшен, связь с инициатором теряется.
3. **Messages Queue → MES API** — потребитель расчёта стоимости может не обработать сообщение из-за ошибки десериализации, таймаута или исключения. Сообщение попадает в DLQ или теряется.
4. **MES API → 3D files storage** — синхронное чтение большого файла может зависнуть, что блокирует поток обработки. Без трейсинга неясно, на каком этапе возникла задержка.
5. **MES API → MES db** — запись статуса `PRICE_CALCULATED` может завершиться неудачей, заказ останется в неопределённом состоянии.
6. **CRM API → MES API (или через очередь)** — подтверждение производства (`MANUFACTURING_APPROVED`) может задержаться из-за ручных действий продавца, но технически сообщение может потеряться в очереди.
7. **MES → MES API (UI)** — дашборд оператора может не отобразить заказ из-за проблем с запросом к БД или кэшем. Важно видеть путь конкретного заказа в системе.
8. **Обратный путь статусов:** цепочка статусов `MANUFACTURING_STARTED`, `COMPLETED`, `PACKAGING`, `SHIPPED`, `CLOSED` проходит через несколько сервисов и очередей. Пропуск одного статуса или сбой приводит к «зависшему» заказу.

**Вывод:** трейсингом должны быть охвачены все API‑сервисы (Shop API, CRM API, MES API), брокер сообщений RabbitMQ (входящие и исходящие сообщения), базы данных и файловое хранилище. Фронтенд-приложения (Internet Shop, CRM, MES) также желательно инструментировать для передачи контекста в API, но в первую очередь покрытие должно быть на бэкенде.

## 2. Данные, которые должны попадать в трейсинг

- **Сквозной идентификатор заказа** (order_id) как бизнес-атрибут для фильтрации всех спанов, связанных с одним заказом.
- **Идентификатор трассировки (trace_id)** – генерируется на входной точке (Shop API при создании заказа или при получении сообщения от партнёра).
- **Идентификатор спана (span_id)** и ссылка на родительский спан.
- **Тип операции:** HTTP-запрос, обработка сообщения RabbitMQ, запрос к БД, обращение к S3, внутренний вызов.
- **Метаданные сервиса:** имя (`shop-api`, `crm-api`, `mes-api`), версия, идентификатор инстанса.
- **Для HTTP-запросов:** URL, HTTP-метод, статус-код ответа, заголовки.
- **Для сообщений RabbitMQ:** имя обменника/очереди, routing key, message_id, заголовки с traceparent.
- **Для базы данных:** тип СУБД, имя таблицы, выполняемая операция (SELECT/INSERT/UPDATE), количество затронутых строк (опционально, без чувствительных данных).
- **Для S3:** имя бакета, операция (PUT/GET), ключ объекта, время отклика.
- **События и ошибки:** исключения с трассировкой стека, коды ошибок.
- **Бизнес-статусы заказа:** какой статус (INITIATED, SUBMITTED, PRICE_CALCULATED и т.д.) был установлен в текущей операции.

## 3. Мотивация

Внедрение распределённого трейсинга критично для компании «Александрит», так как текущие проблемы с «пропадающими» заказами наносят прямой финансовый ущерб (потеря B2B-контрактов, рост жалоб B2C-клиентов). Трейсинг позволит:

- **Сократить время расследования инцидентов (MTTR)** с часов/дней до минут, визуализируя путь конкретного заказа через все сервисы.
- **Точно определить «узкое горлышко»** — где заказ завис или потерялся (очередь, сервис, БД), без необходимости вручную просматривать логи на разных инстансах.
- **Проверить соблюдение SLA** — например, время от отправки заказа до получения цены, или от подтверждения до начала производства.
- **Выявить скрытые аномалии** — повторяющиеся попытки обработки сообщений, длительные блокировки БД, медленные сетевые вызовы к S3.

### Метрики, на которые повлияет внедрение трейсинга (3–5 технических и бизнес-показателей)

1. **Время выполнения сквозного сценария «Заказ → Цена рассчитана» (p50/p95/p99)** — бизнес-метрика, отражающая соблюдение обещанных 2–3 минут.
2. **Доля заказов, успешно дошедших до статуса `CLOSED` без ручного вмешательства** — индикатор качества автоматизации потока.
3. **Количество потерянных сообщений (DLQ)** в разрезе типа заказа (B2B/B2C) — снижение за счёт быстрого обнаружения проблем обработки.
4. **Время нахождения заказа в статусе ожидания подтверждения** — выявляет задержки на стороне продавцов.
5. **Среднее время расследования инцидента (MTTR)** — снижение с нескольких часов до минут.

## 4. Предлагаемое решение

### Технологии и архитектура

В качестве стандарта наблюдаемости принимается **OpenTelemetry**, обладающий широкой поддержкой языков и протоколов, совместимый с различными бэкендами.

**Компоненты трейсинга, которые необходимо внедрить или доработать:**

| Компонент | Действие |
| :--- | :--- |
| **Shop API** (Spring Boot) | Подключить OpenTelemetry Java Agent + Micrometer Tracing. Автоматическое инструментирование HTTP, JDBC, RabbitMQ. Добавить бизнес-атрибут `order_id` и статус заказа во все спаны. |
| **CRM API** (Spring Boot) | Аналогично Shop API, пробрасывать контекст через RabbitMQ и HTTP. |
| **MES API** (C#) | Добавить OpenTelemetry .NET SDK. Инструментировать ключевые операции (расчёт стоимости, доступ к S3, запись в БД, обработка очереди). |
| **Messages Queue (RabbitMQ)** | Использовать встроенную инструментацию OpenTelemetry для продюсеров и консьюмеров, пробрасывая traceparent в заголовках сообщений. |
| **Базы данных (Shop DB, MES DB)** | Инструментация через обёртки драйверов (Otel JDBC driver, OpenTelemetry.SqlClient). |
| **3D files storage (S3)** | Инструментировать через AWS SDK Instrumentation (Java) или ручные спаны в .NET. |
| **OpenTelemetry Collector** (новый компонент) | Развернуть в виде sidecar-контейнера на каждом инстансе или централизованного сервиса. Принимает данные по протоколу OTLP (gRPC/HTTP), выполняет обработку (сэмплирование, маскирование данных) и экспорт в хранилище. |
| **Tracing Backend** (новый компонент) | Использовать Grafana Tempo (или Jaeger), развёрнутую в облаке (Managed Service for Tempo или собственный инстанс). Tempo легко интегрируется с Grafana и Prometheus, поддерживает поиск по атрибутам и метрики из спанов. |

**Схема потоков данных трейсинга:**
- API-сервисы, обогащённые OTel SDK, отправляют телеметрию (spans) в OpenTelemetry Collector по OTLP.
- Collector экспортирует данные в Tempo.
- Пользователи (разработчики, поддержка) через Grafana исследуют трейсы, используя Tempo как датасорс.
