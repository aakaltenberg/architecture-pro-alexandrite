# Архитектурное решение по кешированию

## 1. Мотивация

Операторы производства — ключевые пользователи MES. Их скорость работы напрямую влияет на время выполнения заказов и, как следствие, на удовлетворённость клиентов и выручку компании. Жалобы на медленную загрузку дашборда со списком заказов (с фильтром и пагинацией) свидетельствуют о том, что текущая реализация прямого обращения к базе данных не справляется с ростом количества заказов.

Кеширование позволит:

- **Снизить нагрузку на MES db** — основное узкое место, вызывающее задержки. Повторяющиеся запросы к одним и тем же данным (список заказов по статусам) перестанут нагружать БД.
- **Ускорить отображение дашборда** для операторов, что повысит их производительность и сократит время простоя.
- **Обеспечить плавный рост** без линейного увеличения ресурсов БД при росте числа заказов и пользователей.
- **Улучшить пользовательский опыт** — время загрузки страницы сократится с секунд до миллисекунд.

Кеширование планируется внедрить на уровне **MES API** для данных, отображаемых на дашборде оператора:

- Списки заказов, сгруппированные по статусам и с учётом пагинации.
- Отдельные заказы (детальная информация).
- Возможно, результаты расчёта стоимости (если повторно запрашиваются), но основная цель — дашборд.

## 2. Предлагаемое решение

### 2.1. Тип кеширования: серверное, паттерн Cache-Aside

**Клиентское кеширование** (в браузере) не подходит, потому что:

- Данные быстро меняются (появляются новые заказы), требуется высокая согласованность между операторами.
- Неэффективно при множестве операторов: каждый клиент будет загружать полный список, нагружая сервер, а кеш браузера помогает только при повторных загрузках у одного пользователя.
- Не решает корневую проблему — медленные запросы к БД.

**Серверное кеширование** (на стороне MES API) с использованием **Redis** (in-memory key-value хранилище) — оптимальный выбор. Redis обеспечивает низкую латентность, поддержку сложных структур данных (хеши, сортированные множества) и встроенные механизмы инвалидации.

**Паттерн Cache-Aside** (ленивая загрузка) выбран по следующим причинам:

- **Чтение**: при запросе данных приложение сначала проверяет кеш. Если данные есть — возвращает их, если нет — загружает из БД, сохраняет в кеш и возвращает.
- **Запись**: при изменении данных (новый статус заказа) приложение обновляет БД и явно удаляет соответствующие записи из кеша (инвалидация). Следующее чтение заполнит кеш актуальными данными.

**Почему не другие паттерны:**

| Паттерн | Причина отказа |
|---------|----------------|
| **Write-Through** | Требует синхронной записи в кеш при каждом обновлении. В сценарии MES изменения статусов происходят относительно редко по сравнению с чтениями, но при этом запись в кеш может выполняться дольше из-за дополнительного сетевого вызова. Cache-Aside позволяет отложить заполнение кеша до следующего чтения, что снижает задержку операций записи. Кроме того, Write-Through может привести к засорению кеша данными, которые редко читаются. |
| **Refresh-Ahead** | Предполагает автоматическое обновление записи в кеше до истечения срока жизни. Это полезно для предсказуемо «горячих» данных, но в нашем случае характер обращений непредсказуем (операторы могут запрашивать разные страницы и фильтры). Реализация сложнее, требует фоновых задач и не даёт существенного выигрыша по сравнению с инвалидацией по событию. |

Таким образом, **Cache-Aside** с инвалидацией по ключу обеспечивает баланс между свежестью данных и простотой реализации, минимизируя нагрузку на БД и не усложняя логику записи.

## 2.3. Стратегия инвалидации кеша

**Основная стратегия — инвалидация по ключу при изменении данных.**  
Дополнительно используется **временная инвалидация (TTL)** как защита от устаревания в случае пропуска события.

**Какие ключи удаляются:**

- При изменении статуса заказа `X`:
  - `orders:{id}` — детальная информация о заказе.
  - `orders:status:{old_status}:page:*` — все страницы списка по старому статусу (используем паттерн с `SCAN` или храним префиксы).
  - `orders:status:{new_status}:page:*` — все страницы списка по новому статусу (так как заказ добавится/удалится из списка).

- При создании нового заказа:
  - `orders:status:NEW:page:*` и другие затронутые статусы.

**Почему выбрана инвалидация по ключу, а не другие стратегии:**

| Стратегия | Оценка применимости |
|-----------|---------------------|
| **Временная (TTL)** | Только TTL недостаточно: если установить короткий TTL (секунды), кеш будет часто промахиваться, сводя на нет выигрыш. Длинный TTL приведёт к отображению устаревших данных, что критично для операторов (заказ могут взять несколько раз). |
| **Программная (по событию)** — выбранный вариант | Позволяет немедленно удалять затронутые данные, гарантируя актуальность. Сложность — необходимость корректного перехвата всех событий изменения статуса. Реализуется в MES API, который и так управляет статусами. |
| **Комбинированная (TTL + инвалидация)** — рекомендуемый подход | Мы используем именно её: TTL как fallback (например, 60 секунд) на случай, если событие инвалидации по какой-то причине не сработало. Это предохраняет от бесконечно устаревших данных и упрощает обработку сбоев. |

**Дополнительно:** для ускорения инвалидации групп ключей (по префиксу) можно использовать структуру данных Redis Set, хранящую ключи, сгруппированные по статусу, и удалять их итеративно. Но при небольшом количестве страниц (пагинация) достаточно хранить предсказуемые имена ключей и удалять их по маске с помощью Lua-скрипта или сканирования.

## 3. Дополнительное задание: Сравнение альтернативных решений

Можно рассмотреть два подхода к реализации кеширования дашборда:

| Критерий | Решение 1: Cache-Aside с инвалидацией по ключу (предложенное) | Решение 2: Write-Through кеш (без инвалидации) |
|----------|-------------------------------------------------------------|-----------------------------------------------|
| **Описание** | Кеш заполняется только при чтении; при записи данные удаляются из кеша. | Все записи в БД дублируются в кеш синхронно; чтение всегда из кеша. |
| **Плюсы** | - Минимальная задержка записи (не ждём кеш).<br>- Кеш содержит только востребованные данные.<br>- Простота реализации, не требует транзакций между БД и кешем. | - Данные в кеше всегда актуальны (нет окна неконсистентности).<br>- Чтение всегда быстрое, без промахов. |
| **Минусы** | - Возможно короткое окно неконсистентности между инвалидацией и следующим чтением (решается TTL).<br>- При промахе первый пользователь получает задержку. | - Каждая запись требует двух операций (БД + кеш), увеличивая задержку и нагрузку на сеть.<br>- Кеш может переполняться редко читаемыми данными.<br>- Требуется обработка сбоев кеша при записи. |
| **Вывод** | Лучше подходит для сценария с интенсивным чтением и менее частыми записями, что характерно для дашборда операторов. | Хуже из-за замедления операций изменения статуса, которые критичны для оператора (влияет на отзывчивость интерфейса при взятии заказа). |

**Заключение:** предложенное решение Cache-Aside с событийной инвалидацией и резервным TTL является оптимальным компромиссом между актуальностью данных, производительностью и сложностью реализации.
